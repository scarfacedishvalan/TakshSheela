# Architecture

TakshSheela treats synthetic codebase generation as a compiler pipeline.

Each stage transforms artifacts into more concrete artifacts.

---

# Pipeline

```text
spec → blueprint → fault surface map → canonical codebase
                         ↓                      ↓
                   [scenario spec] → fault injection → assessment
```

A canonical codebase supports multiple scenario variants. Each scenario picks one
fault surface from the fault surface map and injects one fault. The canonical codebase
itself is always correct — it never contains an active fault.

Fault surfaces are declared in the spec as architectural requirements, fulfilled in the
blueprint, and analysed in detail in the fault surface map before any injection occurs.

Code generation takes the approved blueprint directly as input. The blueprint is the
authoritative source for project structure, module layout, data model, logging schema,
and fault surface locations. No intermediate representation stage exists between
blueprint and codebase.

---

## Spec

The spec defines generation intent for a canonical system.

Contains:

* domain and genome (scale, maturity, team style)
* fault families the system must architecturally support
* scenario diversity goals
* operational context
* constraints and exclusions

Spec is a Level 2 artifact — markdown with structured frontmatter.
Human-authored, optionally AI-assisted.

---

## Blueprint

Blueprint is the authoritative architecture artifact for a canonical system.

Contains:

* system overview and history
* subsystem definitions
* project structure (repository layout, module names)
* dependency graph
* runtime lifecycle
* data model
* logging schema (event names, fields, scenario cross-references)
* system invariants
* fault surfaces (one per declared spec surface, with injection description)
* resolved open questions

Blueprint is a Level 2 artifact — markdown with required sections.
Generated by LLM from spec, revised through a critique loop.
Blueprint status must be `approved` before code generation proceeds.

---

## Fault Surface Map

Produced after blueprint approval.

Maps each fault surface declared in the spec to its concrete location in the generated
codebase — specific module, function, and injection type. Enumerates what each injected
fault would look like to a candidate.

The fault surface map is the bridge between canonical system design and scenario
injection. All scenario specs draw their injection targets from it.

Fault surface map is a Level 2 artifact — markdown with structured sections.

---

## Canonical Codebase

Generated Python repository derived directly from the approved blueprint.

Contains:

* packages and modules (layout as specified in blueprint Project Structure section)
* configuration files
* test fixtures
* runtime entrypoints
* structured logging (events as specified in blueprint Logging Schema section)

The canonical codebase is always correct — no active faults. It contains the fault
surfaces described in the blueprint as latent structural characteristics, not as bugs.

Codebases should feel realistic and mid_life where specified. Navigational friction,
v1-era comments, and imperfect naming are intentional. Avoid clean academic code.

---

## Scenario Spec

A scenario spec selects one fault surface from the fault surface map and specifies
the assessment parameters for one scenario variant.

Contains:

* fault surface reference
* fault family and subtype
* injection description
* difficulty
* observability level
* primary capability target
* difficulty calibration across candidate levels

All scenario specs for a problem share the same canonical codebase.
Scenario specs are independent — injecting one does not affect others.

---

## Fault Scenario

A fault-injected variant of the canonical codebase, derived from a scenario spec.

Contains:

* injected codebase (diff from canonical, or input variant for input-dependent faults)
* synthesized observability artifacts (logs, metrics, alerts)
* operational artifacts (incident ticket, runbook excerpt, deploy log)

---

## Assessment Package

Candidate-facing materials derived from a fault scenario.

Contains:

* incident description in operational voice (not exam voice)
* list of available signals
* starting context (what an on-call engineer knows at T+0)
* success criteria (for scoring — not shown to candidate)

---

# Selective Rebuild Philosophy

Lineage is tracked in `lineage.yaml` per problem using content hashes.

Stage invalidation rules:
* spec changes → blueprint invalid → fault surface map invalid → codebase invalid → all scenarios invalid
* blueprint changes → fault surface map invalid → codebase invalid → all scenarios invalid
* fault surface map changes → affected scenario specs invalid
* codebase changes → all scenarios invalid
* scenario spec changes → that scenario's fault scenario and assessment invalid

Lineage schema: `schemas/lineage.schema.json`
